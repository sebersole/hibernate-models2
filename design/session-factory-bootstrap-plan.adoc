= SessionFactory Bootstrap Plan
:toc:

This note sketches the next PoC direction: use the boot-model pipeline to build
enough ORM boot state to instantiate a `SessionFactoryImpl` / `EntityManagerFactory`.

The plan works backward from `org.hibernate.internal.SessionFactoryImpl` because
that constructor is the current runtime boundary.

== Constructor Boundary

`SessionFactoryImpl` currently accepts three inputs:

[source,java]
----
SessionFactoryImpl(
        MetadataImplementor bootMetamodel,
        SessionFactoryOptions options,
        BootstrapContext bootstrapContext)
----

Those inputs are not independent.  They are all products of bootstrap settings,
service-registry construction, source discovery, model categorization, and boot
model binding.

The PoC should make those products explicit rather than recreating the current
inline flow from `EntityManagerFactoryBuilderImpl` and `MetadataBuildingProcess`.

== Some Assumptions

First, a few assumptions we will be working under...

* We no longer need the split in building the EMF from JPA container.  This was something we did for Wild Fly to address a gap in the EE and JPA spec which should no longer be an issue.
* We will need to come back and address other Hibernate projects concerns.  Some of them contributor specialized builders, and we need to make sure those needs are addressed.
* Limit set of bootstrap entries -
** JPA "SE boostrap" - `Persistence`
** JPA "SE boostrap" - `(Hibernate)PersistenceConfiguration`
** JPA "EE bootstrap" - Ultimately `PeristenceUnitDescriptor`
** Legacy `Configuration`, but deprecated in favor of `HibernatePersistenceConfiguration`
** Maybe some form that accepts `AvailableResources` + settings?


== Current Constructor Responsibilities

The constructor still orchestrates a large amount of factory assembly:

* builds the `SessionFactoryServiceRegistry`
* creates the `EventEngine`
* initializes the boot metamodel with the factory
* resolves factory name, JNDI name, UUID, and masked settings
* creates `SqlStringGenerationContext`
* registers integrators and applies integration callbacks
* orders columns and validates the boot metamodel
* primes second-level cache regions
* creates `RuntimeMetamodelsImpl`
* creates `QueryEngineImpl` before runtime metamodel initialization
* creates `SqlTranslationEngine`
* creates and initializes `MappingMetamodelImpl`
* applies fetch profiles after mapping metamodel creation
* creates default session and stateless-session builders
* resolves current-session context
* extracts frequently used services
* scopes `TypeConfiguration` to the completed factory

That sequence is delicate.  The PoC should avoid changing all of it at once, but
it should use named bootstrap products so the handoff into this constructor is
clear.

== Proposed Flow

[source]
----
Bootstrap inputs
    -> ResolvedBootstrapSettings
    -> AvailableResources
    -> CategorizedDomainModel
    -> MetadataImplementor
    -> ResolvedSessionFactoryOptions
    -> SessionFactoryBuildContext
    -> SessionFactoryImpl
----

The immediate goal is not to replace `SessionFactoryImpl`.  The immediate goal
is to produce its three required inputs through the new pipeline.

== Phase Products

=== ResolvedBootstrapSettings

Already prototyped.

Owns early boot settings that affect source collection and mapping processing.
It should not be a dumping ground for all session-factory behavior.

=== AvailableResources

Already prototyped.

Owns normalized source inventory:

* managed class details
* package-info class details
* already-bound XML mapping documents

It should not own XML interpretation, categorization, binding, or settings
resolution.

=== CategorizedDomainModel

Already prototyped.

Owns interpreted domain metadata before binding to ORM's boot mapping model.
This is currently where XML source interpretation lives because there is no
standalone prepared-source artifact.

=== MetadataImplementor

This is the first major bridge into existing ORM runtime bootstrap.

The PoC binder should produce a normal ORM boot mapping model that can satisfy
`SessionFactoryImpl` expectations:

* persistent classes
* collections
* database / namespace / table state
* type registrations and converter registrations
* filters, fetch profiles, named queries, imports, and global registrations
* cache region configuration

This is the main proving ground for the binding work.

=== ResolvedSessionFactoryOptions

This is proposed, not yet prototyped.

`SessionFactoryOptions` is very large and mixes resolved values, service
references, JPA/native defaults, and raw setting interpretation.  A resolver
similar to `BootstrapSettingsResolver` would make the factory-build handoff more
explicit.

Potential responsibilities:

* apply JPA-vs-native defaults
* resolve factory name, UUID, observer list, interceptor, statement observer,
  statement inspector, and session defaults
* resolve tenant, cache, JDBC, transaction, statistics, query, and compliance
  options
* preserve a raw settings map for components that still require one
* expose the `StandardServiceRegistry` used for factory construction

This can initially adapt to ORM's existing `SessionFactoryOptions` interface.
The design point is to move option resolution out of the final constructor path.

=== SessionFactoryBuildContext

This is proposed, not yet prototyped.

`BootstrapContext` currently spans early source collection, model building, and
runtime metamodel creation.  That makes it easy for early bootstrap concepts to
leak into late factory construction.

A later build context could hold only the values needed after boot model binding:

* `BootstrapContext` or a narrowed replacement for runtime model creation
* `MetadataImplementor`
* resolved session-factory options
* service registry
* `TypeConfiguration`
* resolved settings map
* `SqlStringGenerationContext`

This does not need to replace ORM's `BootstrapContext` immediately.  It can
start as a local planning object used by the PoC factory bootstrap.

== Improvements Worth Prototyping

=== Resolve SessionFactoryOptions Explicitly

Build a resolver that produces an adapter implementing ORM's
`SessionFactoryOptions` from resolved settings and services.

This makes the PoC path testable without forcing all option interpretation to
remain hidden in existing builders.

=== Split Early BootstrapContext From Late Build Context

Keep `BootstrapContext` for early model-building concerns.  Introduce a late
`SessionFactoryBuildContext` only when preparing to instantiate the factory.

The aim is to make the handoff visible:

[source]
----
MetadataImplementor + ResolvedSessionFactoryOptions + BootstrapContext
    -> SessionFactoryBuildContext
    -> SessionFactoryImpl
----

=== Make Runtime Model Creation a Named Phase

`SessionFactoryImpl` currently performs runtime metamodel creation inline.  The
sequence is constrained by query-engine and SQL-translation dependencies.

For now, leave that logic in `SessionFactoryImpl`, but document it as a future
phase:

[source]
----
validated MetadataImplementor
    -> RuntimeMetamodels
    -> QueryEngine / SqlTranslationEngine
    -> fetch profile application
----

=== Treat Integrators as an Explicit Phase

Integrators currently run inside `SessionFactoryImpl` before boot metamodel
ordering, validation, cache priming, and runtime metamodel creation.

That phase should eventually be visible:

[source]
----
bound boot model
    -> integrators applied
    -> boot model ordered and validated
    -> runtime model created
----

=== Avoid Requiring an Incomplete SessionFactory

`SessionFactoryImpl` builds the session-factory service registry by passing
`this` before the factory is fully initialized.

That may remain unavoidable in the short term, but it is a useful design smell.
An eventual factory-build handle could allow services that need factory identity
to avoid depending on a partially initialized `SessionFactoryImpl`.

. Find concrete consumers in ORM.

=== BootstrapContext

Initial inspection target: `SessionFactoryImpl` and the first pass-through level
(`Integrator` and `RuntimeModelCreationContext` consumers).

Direct `SessionFactoryImpl` usage:

* `bootstrapContext.getTypeConfiguration()` is read at constructor start and
  stored as the factory `TypeConfiguration`.
* the full `BootstrapContext` is passed to `Integrator#integrate`.
* the full `BootstrapContext` is passed through
  `RuntimeModelCreationContext#getBootstrapContext()` during mapping/JPA
  runtime model creation.

This means the constructor itself does not appear to need the whole contract.
The broad need comes from extension and runtime-model pass-through.

[cols="1,2,2,2", options="header"]
|===
| Consumer area
| Observed access
| Likely narrower dependency
| Notes

| `SessionFactoryImpl`
| `getTypeConfiguration()`
| `TypeConfiguration`
| Direct constructor need.  This can be made explicit in a later
  `SessionFactoryBuildContext`.

| Integrators
| `Integrator#integrate(Metadata, BootstrapContext, SessionFactoryImplementor)`
| Compatibility adapter for existing `Integrator` contract
| This is an extension compatibility concern.  Existing integrators may use any
  `BootstrapContext` method, so narrowing probably requires a staged SPI change
  or adapter.

| Runtime model creation
| `RuntimeModelCreationContext#getBootstrapContext()`
| Runtime model creation context with explicit values
| Known downstream uses include class-loading access, bean registry/access,
  representation strategy selection, metadata building options, service registry,
  and type/configuration access.

| Runtime JPA metamodel
| `runtimeModelCreationContext.getBootstrapContext().getClassLoaderService()`
| `ClassLoaderService` or class-loading access
| Example direct use in `JpaMetamodelImpl`.

| Runtime representation strategies
| `getClassLoaderAccess()`, `getManagedBeanRegistry()`,
  `getRepresentationStrategySelector()`
| representation strategy creation services
| Examples include POJO/entity/embeddable representation strategy construction.

| Persister/runtime mapping construction
| `getMetadataBuildingOptions()`, `getServiceRegistry()`
| mapping/runtime creation options and service access
| Examples include collection/entity persister initialization and multi-table
  strategy support.
|===

Preliminary conclusion:

`BootstrapContext` should not be treated as a single required constructor
dependency.  For the PoC, keeping it as an adapter may be acceptable for
integrator compatibility and current runtime model creation, but the design
target should be to split it into:

* explicit direct constructor inputs such as `TypeConfiguration`;
* a narrower runtime model creation context;
* an integrator compatibility adapter until the extension SPI can be revisited;
* early boot-only services that should not reach `SessionFactoryImpl`.

==== Usage Buckets

The specific `BootstrapContext` uses still need to be classified.  The intended
classification is:

Early boot only::
Should not reach `SessionFactoryImpl`.
+
Likely examples include source discovery, XML processing, metadata binding
helpers, `ModelsContext`, temporary classloader access, and scanning artifacts.

Runtime model creation::
Should move onto, or remain available through, a narrower
`RuntimeModelCreationContext`.
+
Likely examples include `ClassLoaderService`, `ManagedBeanRegistry`,
representation strategy selection, and any mapping/runtime options that are
still proven necessary.

Integrator compatibility::
May require a legacy `BootstrapContext` adapter in the short term.
+
`Integrator#integrate(Metadata, BootstrapContext, SessionFactoryImplementor)` is
the strongest compatibility reason to preserve a `BootstrapContext` view until a
new extension SPI exists.

This classification has not been completed yet.  Each observed use should be
assigned to one of these buckets before introducing a replacement contract.

==== PoC Decision

The PoC should not treat `BootstrapContext` as a primary bootstrap product.
Where existing ORM integration points still require the legacy contract, the PoC
may create or carry an adapter.  New PoC phases should request explicit values
instead of accepting `BootstrapContext`.

Follow-up questions:

* Which `BootstrapContext` methods are actually used by built-in integrators,
  especially Bean Validation?
* Can `RuntimeModelCreationContext` expose the needed values directly and
  deprecate `getBootstrapContext()`?
* Does runtime model creation still need `MetadataBuildingOptions`, or only a
  smaller set of resolved mapping/runtime options?
* Can `TypeConfiguration` be produced/scoped without retaining an early
  `BootstrapContext` reference?

=== InFlightMetadataCollector

Initial inspection target: PoC orchestration and binding code.

`InFlightMetadataCollector` is currently the mutable ORM metadata target.  The
PoC uses it both directly and indirectly through `MetadataBuildingContext`.
Unlike `MetadataBuildingContext`, most collector uses are not merely accidental
service access; many are real writes to the ORM boot metadata model that is
returned as `MetadataImplementor`.

Observed usage in the PoC:

[cols="1,2,2,2", options="header"]
|===
| Consumer area
| Observed access
| Likely narrower dependency
| Notes

| `MetadataResolver`
| creates `InFlightMetadataCollectorImpl`; after binding calls
  `orderColumns(false)` and `validate()` on the metadata result
| metadata target factory plus finalization step
| This is the clearest orchestration-level dependency.  Longer term this could
  become a `MetadataBuildTarget` / finalizer pair, but for now it is the bridge
  that proves the PoC can produce ORM metadata.

| `DomainModelCategorizer`
| reads `getPersistenceUnitMetadata()`, `getConverterRegistry()`, and
  `getDatabase()` from the collector before constructing `CategorizationContext`
| persistence-unit metadata, converter registry, database
| The categorization model no longer sees the collector.  Remaining reads are
  entry-point plumbing while `DomainModelCategorizer` still accepts
  `MetadataBuildingContext`.

| `BindingStateImpl`
| reads `getDatabase()` during construction; writes `addEntityBinding`,
  `addMappedSuperclass`, and `addFilterDefinition`
| database, managed-type registration target, filter-definition target
| Partly done: `BindingState` now localizes these writes.  Entity bindings still
  publish immediately because property-reference binding validates against the
  collector during the same binding run; mapped-superclass, collection, import,
  filter, and global-registration writes use the explicit final bridge.

| `BindingCoordinator` global registrations
| writes identifier generators, named entity graphs, converters, Java/JDBC type
  registrations, user/composite user types, collection type registrations, and
  embeddable instantiators
| global-registration target
| Done for non-query registrations: these now write to `BindingState` and are
  applied to the collector by the final bridge.  Query registrations still use
  ORM's legacy query binders.

| Query binding
| `QueryBinder.bindQuery`, `bindNativeQuery`, and
  `bindNamedStoredProcedureQuery` require `MetadataBuildingContext`
| query registration adapter
| This likely waits for upstream integration.  ORM's query binders currently
  consume the legacy context and write to the collector internally.

| Table binding
| `addTable` and `addDenormalizedTable`; table naming callbacks also expose
  `MetadataBuildingContext`
| database namespace/table factory
| This is a core collector responsibility today.  A narrow table factory would
  be cleaner, but ORM's `Table` creation and namespace registration currently
  live on the collector.

| Collection binding
| `addCollectionBinding`
| collection metadata target
| Done: collection binders now record collection bindings in `BindingState`;
  the collector remains the final storage target.

| Entity and mapped-superclass imports
| `addImport`
| import registration target
| Done: imports now record in `BindingState` and publish during the final bridge.

| Unique property references
| `addUniquePropertyReference`
| property-reference registration target
| Still an immediate collector side channel, but now localized behind
  `BindingState#addUniquePropertyReference`.  It cannot wait for the final
  bridge because later binding phases consume it during the same binding run.
  Upstream should ideally grow a typed property-reference API.

| Tenant-id binding
| reads `getTypeConfiguration()` and `getFilterDefinition()`; may write
  `addFilterDefinition`
| type configuration plus filter-definition registry
| Partly done: type configuration and filter lookup/write now go through
  `BindingState` so tenant-id validation sees pending filter definitions.

| `BindingOptionsImpl`
| reads `getDatabase().getJdbcEnvironment()`
| JDBC environment
| This is settings/defaults support, not a conceptual collector dependency.
|===

Preliminary conclusion:

`InFlightMetadataCollector` is not just an overly broad service locator.  In the
current bridge it is the mutable ORM metadata product.  The healthier split is:

* keep the collector as the short-term final metadata target;
* hide simple writes behind `BindingState` or a future boot metadata target
  (`addEntityBinding`, `addMappedSuperclass`, `addCollectionBinding`,
  `addImport`, filter definitions, global registrations);
* isolate table/namespace creation separately if upstream ORM is changed;
* leave query binding and mapping-object construction on the legacy adapter until
  those upstream APIs can be changed;
* keep final ordering/validation as an explicit orchestration step.

This audit suggests that removing collector exposure locally is useful only when
it improves binder shape without fighting ORM's mapping constructors.  The PoC
now uses `BindingState#applyMetadataRegistrations` as the explicit bridge for
simple metadata writes that can wait until the end of binding.  Entity bindings
and unique property references remain immediate but are localized behind
`BindingState` because later binding phases depend on the collector side effects.
Table creation and query binding still need upstream work before a local rewrite
is worthwhile.

=== MetadataBuildingContext

Initial inspection target: PoC categorization, binding, and orchestration code.

`MetadataBuildingContext` is not currently part of the `SessionFactoryImpl`
constructor boundary, but it is a major reason the PoC still creates an ORM
`BootstrapContext`, `MetadataBuildingOptions`, and `InFlightMetadataCollector`.
It is also passed into many ORM mapping model constructors.

Observed usage in the PoC:

[cols="1,2,2,2", options="header"]
|===
| Consumer area
| Observed access
| Likely narrower dependency
| Notes

| `MetadataResolver`
| Creates `AvailableResourcesContext` from
  `metadataBuildingContext.getBootstrapContext().getModelsContext()` and
  `getServiceRegistry()`; reads `getMetadataCollector()` after binding.
| `ModelsContext`, `ServiceRegistry`, final metadata result
| Source collection no longer needs the whole context.  This is already mostly a
  convenience leak from current orchestration.

| `DomainModelCategorizer`
| `getBootstrapContext()`, `getMetadataCollector().getPersistenceUnitMetadata()`,

| `CategorizationContextImpl`
| direct constructor inputs: persistence-unit metadata, effective defaults,

| explicit categorization collaborators

| `BindingStateImpl`

  `getMetadataCollector().getDatabase()` and `JdbcServices` from
  `getBootstrapContext().getServiceRegistry()`

  `getMetadataBuildingContext()` as an escape hatch.

  `getBootstrapContext().getServiceRegistry().getService(ConfigurationService)`,
  `getMetadataCollector().getDatabase().getJdbcEnvironment()`

| `BindingContextImpl`

Methods not yet observed as central in the PoC audit:

* `getTypeDefinitionRegistry()`
* `getCurrentContributorName()`
* `getTemporalTableStrategy()`
* `getAuditStrategy()`

They may still be used indirectly by ORM mapping objects during validation or
runtime model creation, so this is not a final removal claim.

Preliminary conclusion:

`MetadataBuildingContext` is mainly a legacy aggregation point for three
different concerns:

* services/options/defaults needed during categorization and binding;
* the mutable metadata collector / binding target;
* ORM mapping object construction APIs that require the legacy context type.

The PoC should not treat it as the conceptual binding contract.  The likely
direction is:

* `CategorizationContext` is now independent of `MetadataBuildingContext`;
* continue growing `BindingState` into the typed mutable binding target;
* make `BindingOptions` / `BindingContext` consume explicit values;
* retain a small ORM-compatible `MetadataBuildingContext` adapter only where
  mapping object constructors or collector APIs still require it.

This audit has not been completed yet.  The next useful step is to list the
remaining `BindingState#getMetadataBuildingContext()` callers and decide whether
each one is:

* a real ORM mapping-constructor requirement;
* a collector side effect that should move to `BindingState`;
* a settings/defaults lookup that should move to `BindingOptions` or
  `BindingContext`;
* or a categorization concern that should move earlier.

Follow-up questions:

* Can `BindingState` hide `MetadataBuildingContext` from most binders?
* Which collector writes are missing typed methods on `BindingState`?
* Can ORM mapping constructors be isolated behind local factory methods so the
  legacy context dependency is contained?
* Can `CategorizationContext` be built directly from resolved settings,
  `AvailableResources`, and metadata target services?

=== MetadataBuildingOptions

Initial inspection target: PoC orchestration and binding code.

`MetadataBuildingOptions` is still created by ORM's `MetadataBuilderImpl` and
threaded through `MetadataBuildingContext`.  In the PoC it acts mostly as a
container for resolved mapping options and naming strategies, not as a mutable
metadata target.

Observed usage in the PoC:

[cols="1,2,2,2", options="header"]

| Observed access
| Likely narrower dependency

Preliminary conclusion:

`MetadataBuildingOptions` is less problematic than `MetadataBuildingContext` and
less central than `InFlightMetadataCollector`: it mostly contains values we do
need, but under a broad legacy container.

The likely direction is:

* keep creating it while the PoC reuses ORM's metadata builder/collector/context
  bridge;
* move local consumers toward explicit values already available from
  `BindingContext` and `BindingOptions`;
* grow a named mapping-settings contract for joined-inheritance discriminator
  behavior, implicit catalog/schema, global quoting, naming strategies, and
  shared cache mode;
* keep `MetadataBuildingOptions` as an adapter detail until ORM mapping
  constructors and XML/query processors no longer require the legacy context.

Near-term local cleanup candidates:

* have `TableBinder` use `BindingContext#getImplicitNamingStrategy()` and
  `getPhysicalNamingStrategy()`;
* have `EntityTypeBinder` use `BindingContext#getSharedCacheMode()` and
  `getPhysicalNamingStrategy()`;

=== SessionFactoryServiceRegistry

Not audited yet.

== Initial Prototype Target

The first implementation target should be deliberately narrow:

[source]
----
AvailableResources
    -> DomainModelCategorizer
    -> BindingCoordinator
    -> MetadataImplementor
    -> existing SessionFactoryOptions adapter
    -> existing BootstrapContext
    -> new SessionFactoryImpl(...)
----

Success criteria:

* a small annotated model can bootstrap a usable `SessionFactory`
* the PoC binder produces a `MetadataImplementor` accepted by the constructor
* option resolution is explicit enough to see which settings are still missing
* no new abstraction is added unless it removes an actual handoff ambiguity

== Current PoC Slice

The current metadata target resolver is:

* link:../src/main/java/org/hibernate/boot/settings/BootstrapSettingsResolver.java[`BootstrapSettingsResolver`]
* link:../src/main/java/org/hibernate/boot/orchestration/MetadataResolver.java[`MetadataResolver`]
* link:../src/main/java/org/hibernate/boot/orchestration/ResolvedMetadata.java[`ResolvedMetadata`]

Settings resolution is a separate target:

[source]
----
configuration values / JPA inputs
    -> BootstrapSettingsResolver
    -> ResolvedBootstrapSettings
----

`MetadataResolver#resolve` owns only the metadata target phase order:

[source]
----
ResolvedBootstrapSettings + BootstrapSourceContributions + ServiceRegistry
    -> MetadataBuildingContext
    -> AvailableResources
    -> DomainModelCategorizer
    -> BindingCoordinator
    -> order columns
    -> validate
    -> ResolvedMetadata
----

`ResolvedMetadata` carries the finalized `MetadataImplementor`, the
`CategorizedDomainModel`, the `BindingState` used as the local metadata-write
bridge, and the `ResolvedBootstrapSettings`.

Later slices should add one resolver per gross target rather than growing a
single SessionFactory-wide orchestrator.

== Open Questions

* Should `ResolvedSessionFactoryOptions` be a record plus adapter, or should the
  adapter itself be the resolved object?
* Which settings belong in `ResolvedBootstrapSettings` versus
  `ResolvedSessionFactoryOptions`?
* Can `BootstrapContext` be narrowed before integration, or should the PoC simply
  reuse ORM's current implementation until the factory path works?
* Should XML source interpretation remain in categorization internals, or should
  a prepared-source phase emerge if more source interpretation steps appear?
* Can service-registry creation be moved before `SessionFactoryImpl` without
  breaking services that expect a concrete factory reference?
